Docker Submission Guide

Step-by-step instructions for preparing, testing, and submitting your final Docker container. Follow these steps in order. Test-set evaluation is manual and offline: there is no interim feedback, so a container that fails to run will not be flagged until after the deadline has closed.

Deadline update: the final Docker submission deadline has been extended from 1 August 2026 to 20 August 2026 to allow time to publish the full container harness specification referenced in Step 4 below. This does not affect the results release date of 27 September 2026. Steps marked coming soon will be completed once the harness specification is published; all other steps below are final and can be acted on now.
1
Confirm your Synapse account is a Certified User. This is a Synapse-wide requirement, separate from anything specific to this challenge, and it silently blocks Docker push and submission with no explanatory error shown anywhere in the challenge interface.
  • Go to your Synapse account: click your profile icon → Settings → Trust & Credentials
  • If it shows Certification: Incomplete, click Take The Certification Quiz (roughly 15 minutes, ~15 questions on Synapse data governance)
  • Submit the quiz and confirm the page now shows Certification: Complete

Do this before attempting anything else in this guide. An uncertified account can be a member of the Submitters team with "Can submit" access on every queue and still be unable to push an image or submit, with no error message explaining why. If you get stuck on any later step, check this first.

2
Submission rights on all three evaluation queues are granted to members of the EchoRisk-MICCAI Submitters team. Membership is added automatically for the team member whose Synapse User Profile URL was supplied on the official EchoRisk-MICCAI 2026 Team Registration form. You do not need to request access if you are that person.

Only one designated contact per team is granted submission access, regardless of team size. Your team's official entry is whichever container that person uploads; there is no need or ability for multiple members to submit independently. To change who your designated submitter is, email echorisk.miccai@gmail.com with the new Synapse Profile URL.

If you cannot complete Step 5 below, it is almost always one of the following:
  • Your account is not yet a Certified User (Step 1)
  • You are not the team member who filled in the registration form, so you were never added to the Submitters team
  • The registration form was submitted with a missing or incorrect Synapse Profile URL
  • You are a teammate other than the designated submitter — access is not extended automatically to additional members; email us to change the designated contact instead
In any of these cases, email echorisk.miccai@gmail.com with your team name and the Synapse username to be set as your designated submitter. Requests are turned around within one business day, but should be sent well ahead of the deadline rather than on the day it closes.
3
Identify the evaluation queue for your task. Each task has its own dedicated queue; you will select the correct one by name at the final step of submission (Step 6), not by visiting the Challenge page first. Note the exact queue name for your task(s) now:
  • Task 1 — Cardiac Function Estimation: EchoRisk-MICCAI 2026 - Docker Submission (Task 1)
  • Task 2 — Assessment of LV Dysfunction: EchoRisk-MICCAI 2026 - Docker Submission (Task 2)
  • Task 3 — Early Prediction of Therapy-Induced Cardiotoxicity: EchoRisk-MICCAI 2026 - Docker Submission (Task 3)
These are the final test-set queues only. There is no validation leaderboard or interim feedback associated with them; submitted containers are run against the hidden test set by the organisers after the deadline closes. The Challenge project's Wiki page lists these queues for reference, but there is no submission action available there; submission is always initiated from your own project, described in Step 6.
4
Prepare your predictions and container. This step has two parts. The prediction format is final and documented now; the container harness contract is not yet published.

Prediction format (confirmed). The scoring code in the evaluation repository defines the exact CSV your model must produce per task, keyed by (patient_id, timepoint), covering every exam in the ground-truth set:

github.com/EchoRisk-MICCAI/echorisk-benchmark/tree/master/evaluation
  • Task 1: lvef_pred, float in [0, 100]
  • Task 2: dysfunction_pred, calibrated probability in [0, 1], not a hard label
  • Task 3: cardiotoxicity_pred, calibrated probability in [0, 1], baseline (T1) only
Run the official scorer locally on the validation set before packaging, so your offline numbers match what the leaderboard will compute:
pip install -e "./evaluation[test]"
echorisk-score --task 1 --predictions preds.csv --ground-truth gt.csv

The scorer enforces schema strictly: missing exams, out-of-range values, duplicate (patient_id, timepoint) keys, and missing columns are all rejected. A clean local run is a genuine correctness signal, not just a formality.

Container harness contract — coming soon. The entrypoint invocation, input mount path, output file path, and base image/GPU requirements for the containerised evaluation harness are not yet published here. This section will be completed before the new submission deadline above. Do not finalise your Dockerfile against assumed paths; check back here first.

5
Test offline before submitting. Per Challenge Rule 6, containers must run fully automatically with no manual intervention, no internet access, and no external API calls at inference time. Validate this locally before submission:
docker run --rm --network none --gpus all \
  -v /path/to/local/test_case:/input:ro \
  -v /path/to/local/output:/output \
  your-image:tag

--network none reproduces the offline evaluation environment. If your container depends on a runtime download (model weights, tokeniser files, API calls), it will fail evaluation. The mount paths shown above are illustrative only; the exact paths the harness uses will be confirmed in Step 4 once the harness specification is published.

6
Submit your container. Docker submissions are not made from a button on the Challenge page. The Challenge page never shows a "Submit" option for Docker queues; submission is initiated from your own Synapse project instead. Only your team's designated submitter (Step 2) can complete this. Follow these steps in order:
  1. Create (or use an existing) Synapse Project that you own, this is where your image will be pushed to, not the challenge project itself
  2. Get your Docker registry credentials: go to Account Settings → Personal Access Tokens, create a new token with at least View, Download, Modify scope, and copy it (shown only once)
  3. From your terminal, log in to Synapse's Docker registry:
    docker login docker.synapse.org
    using your Synapse username and the Personal Access Token as the password
  4. Tag and push your image, using your own project's SynID (found on your project page, format synXXXXXXXX):
    docker tag your-image:tag docker.synapse.org/synXXXXXXXX/your-image:tag
    docker push docker.synapse.org/synXXXXXXXX/your-image:tag
  5. Go to your project in Synapse, click the Docker tab, refresh if needed, your pushed image should now be listed there
  6. Click on the image entry, then click Docker Repository Tools → Submit Docker Repository to Challenge
  7. In the dialog, select the evaluation queue matching your task (Step 3), and confirm

If the push fails with "insufficient_scope" or "access denied": this is almost always the Step 1 certification requirement, not a token or permissions problem. Confirm Trust & Credentials shows Certification: Complete, then retry the push.

You may resubmit at any point before the deadline to update your container; only your last submission is evaluated, consistent with Challenge Rule 2.

7
Deadline and what happens after. The submission window closes on 20 August 2026, extended from the original 1 August 2026 date to allow the container harness specification (Step 4) to be published in time for teams to act on it. After the deadline, organisers run all final containers against the hidden test set. No test-set metrics are visible to participants before or during this process. Results are still released on 27 September 2026 during the MICCAI Annual Meeting; the extension does not affect this date.

// Q&A

Submission FAQ

Docker push fails with "insufficient_scope" or "access denied"
This is the single most common blocker and is almost never a token or team-permissions issue. Go to Synapse Account Settings → Trust & Credentials and confirm your account shows Certification: Complete. If it shows Incomplete, click Take The Certification Quiz and retry the push immediately after. See Step 1 above.
I don't see a Submit button on the Challenge page
Correct, there isn't one for Docker submissions. The Challenge page only ever shows evaluation queue descriptions, not a submit action. Submission is initiated from your own Synapse project via Docker Repository Tools → Submit Docker Repository to Challenge, after pushing your image there. See Step 6 above for the full sequence.
I accepted the Synapse Team invitation. Why can't I submit?
Submission rights come from a separate team, EchoRisk-MICCAI Submitters, populated automatically from the Synapse Profile URL given on your registration form. See Step 2 above for the most common reasons this hasn't happened (including certification, Step 1, and the one-designated-submitter policy below), then contact echorisk.miccai@gmail.com with your team name and Synapse username(s).
Can more than one team member get submission access?
No. Docker submission access is granted to one designated contact per team, matching the Synapse Profile URL on your Team Registration Form. Only one member needs access, since your team's official entry is whichever container that person uploads on the team's behalf. To change who this is, email echorisk.miccai@gmail.com with the new Synapse Profile URL.
Is there a template or example repository to follow?
The evaluation repository defines the final prediction CSV format and the official local scorer now. The container harness specification (entrypoint, mount paths, base image) is not yet published; see Step 4.
Can I submit different containers for different tasks?
Yes. Each task has its own evaluation queue (see Step 3) and is evaluated independently. Submit only to the queue(s) for the task(s) you are entering.
Why was the deadline extended?
To give teams enough time to act on the container harness specification once it is published (Step 4), rather than packaging against assumed paths. The extension does not affect the results release date of 27 September 2026.
Will I get feedback if my container fails?
No. These are hidden test-set queues with no interim leaderboard or feedback. This is why offline testing before submission (Step 5) is essential; a container that fails to run cannot be corrected after the deadline closes.
