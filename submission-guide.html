Docker Submission Guide

Step-by-step instructions for preparing, testing, and submitting your final Docker container. Follow these steps in order. Test-set evaluation is manual and offline: there is no interim feedback, so a container that fails to run will not be flagged until after the deadline has closed.

Deadline update: the final Docker submission deadline has been extended from 1 August 2026 to 20 August 2026 to allow time to publish the full container harness specification referenced in Step 3 below. This does not affect the results release date of 7 October 2026. Steps marked coming soon will be completed once the harness specification is published; all other steps below are final and can be acted on now.
1
Submission rights on all three evaluation queues are granted to members of the EchoRisk-MICCAI Submitters team. Membership is added automatically for the team member whose Synapse User Profile URL was supplied on the official EchoRisk-MICCAI 2026 Team Registration form. You do not need to request access if you are that person.

If you cannot see a Submit button on the evaluation queues described in Step 2, it is almost always one of the following:
  • You are not the team member who filled in the registration form, so you were never added to the Submitters team
  • The registration form was submitted with a missing or incorrect Synapse Profile URL
  • Your team wants additional members beyond the original registrant to be able to submit
In any of these cases, email echorisk.miccai@gmail.com with your team name and the Synapse username(s) to be added to Team 3603440. Requests are turned around within one business day, but should be sent well ahead of the deadline rather than on the day it closes.
2
Locate the evaluation queue for your task. All three evaluation queues are accessed from the Challenge tab on the EchoRisk-MICCAI Synapse project, not the project's default landing page. Each task has its own dedicated queue; submit only to the queue(s) matching the task(s) you are entering.
  • Task 1 — Cardiac Function Estimation: Evaluation Queue 9620025
  • Task 2 — Assessment of LV Dysfunction: Evaluation Queue 9620026
  • Task 3 — Early Prediction of Therapy-Induced Cardiotoxicity: Evaluation Queue 9620027
These are the final test-set queues only. There is no validation leaderboard or interim feedback associated with them; submitted containers are run against the hidden test set by the organisers after the deadline closes.
3
Prepare your predictions and container. This step has two parts. The prediction format is final and documented now; the container harness contract is not yet published.

Prediction format (confirmed). The scoring code in the evaluation repository defines the exact CSV your model must produce per task, keyed by (patient_id, timepoint), covering every exam in the ground-truth set:

github.com/EchoRisk-MICCAI/echorisk-benchmark/tree/master/evaluation
  • Task 1: lvef_pred, float in [0, 100]
  • Task 2: dysfunction_pred, calibrated probability in [0, 1], not a hard label
  • Task 3: cardiotoxicity_pred, calibrated probability in [0, 1], baseline (T1) only
Run the official scorer locally on the validation set before packaging, so your offline numbers match what the leaderboard will compute:
pip install -e "./evaluation[test]"
echorisk-score --task 1 --predictions preds.csv --ground-truth gt.csv

The scorer enforces schema strictly: missing exams, out-of-range values, duplicate (patient_id, timepoint) keys, and missing columns are all rejected. A clean local run is a genuine correctness signal, not just a formality.

Container harness contract — coming soon. The entrypoint invocation, input mount path, output file path, and base image/GPU requirements for the containerised evaluation harness are not yet published here. This section will be completed before the new submission deadline above. Do not finalise your Dockerfile against assumed paths; check back here first.

4
Test offline before submitting. Per Challenge Rule 6, containers must run fully automatically with no manual intervention, no internet access, and no external API calls at inference time. Validate this locally before submission:
docker run --rm --network none --gpus all \
  -v /path/to/local/test_case:/input:ro \
  -v /path/to/local/output:/output \
  your-image:tag

--network none reproduces the offline evaluation environment. If your container depends on a runtime download (model weights, tokeniser files, API calls), it will fail evaluation. The mount paths shown above are illustrative only; the exact paths the harness uses will be confirmed in Step 3 once the harness specification is published.

5
Submit via the evaluation queue. Open the queue for your task on Synapse and use the Submit button to push your image. You may resubmit at any point before the deadline to update your container; only your last submission is evaluated, consistent with Challenge Rule 2.
6
Deadline and what happens after. The submission window closes on 20 August 2026, extended from the original 1 August 2026 date to allow the container harness specification (Step 3) to be published in time for teams to act on it. After the deadline, organisers run all final containers against the hidden test set. No test-set metrics are visible to participants before or during this process. Results are still released on 27 September 2026 during the MICCAI Annual Meeting; the extension does not affect this date.

// Q&A

Submission FAQ

I accepted the Synapse Team invitation. Why can't I submit?
Submission rights come from a separate team, EchoRisk-MICCAI Submitters, populated automatically from the Synapse Profile URL given on your registration form. See Step 1 above for the three most common reasons this hasn't happened, then contact echorisk.miccai@gmail.com with your team name and Synapse username(s).
Is there a template or example repository to follow?
The evaluation repository defines the final prediction CSV format and the official local scorer now. The container harness specification (entrypoint, mount paths, base image) is not yet published; see Step 3.
Can I submit different containers for different tasks?
Yes. Each task has its own evaluation queue (see Step 2) and is evaluated independently. Submit only to the queue(s) for the task(s) you are entering.
Why was the deadline extended?
To give teams enough time to act on the container harness specification once it is published (Step 3), rather than packaging against assumed paths. The extension does not affect the results release date of 27 September 2026.
Will I get feedback if my container fails?
No. These are hidden test-set queues with no interim leaderboard or feedback. This is why offline testing before submission (Step 4) is essential; a container that fails to run cannot be corrected after the deadline closes.
